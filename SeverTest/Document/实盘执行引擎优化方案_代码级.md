# 实盘执行引擎优化方案（代码级）

> 基于 `RealTimeStrategyEngine` 全链路代码审查，给出 4 个阶段的具体改造方案。
> 每阶段独立可验证，逐步推进。

---

## 0. 当前热路径开销拆解

以 `HandleTask → ExecuteStrategyTask → ExecuteLogic → EvaluateChecks` 为主链路，
对一个**典型策略**（2 容器 × 2 组 × 5 条件）执行一次，堆分配情况如下：

| 位置 | 分配内容 | 每策略次数 |
|------|---------|-----------|
| `HandleTask:287` | `new StrategyRunTrace()` | 1（全局共享） |
| `HandleTask:288` | `new StrategyRunMetrics()` + 2 个 `ConcurrentDictionary` | 1（全局共享） |
| `HandleTask:285,294,308,318,362` | `Stopwatch.StartNew()` × 5 | 1 全局 + 1 每策略 |
| `ExecuteStrategyTask:361` | `new StrategyExecutionContext(...)` | 1 |
| `ExecuteBranch:500` | `new List<ConditionContainer>()` 备选 | 0~1 |
| `ExecuteBranch:502` | `new List<ConditionEvaluationResult>()` aggregated | 1 每分支 |
| `ExecuteBranch:515` | `new List<ConditionEvaluationResult>()` per-container | 2 |
| `ExecuteBranch:516` | `$"{stage}[{i}]"` 字符串插值 | 2 |
| `EvaluateChecks:643` | `new List<ConditionGroup>()` 备选 | 0~1 |
| `EvaluateChecks:654-657` | 4 × `new List<StrategyMethod>()` per group | 4 × 4 = 16 |
| `EvaluateChecks:685` | `_logger.LogDebug(...)` 7 参数装箱 | 每条件 1 次（≈20） |
| `EvaluateEntryFilters:440` | `new List<ConditionEvaluationResult>()` | 1 |
| `EvaluateEntryFilters:441` | `$"{stage}.Filter"` 字符串插值 | 1 |
| `QueuedStrategyActionExecutor:46-74` | `new StrategyActionTask` + LINQ `.Select().ToList()` | 0~1 |

**合计：每策略约 25~30 次堆分配 + 100~140 次值类型装箱。**
71 策略 × 30 = **~2100 次堆分配/tick**，GC 压力显著。

## Phase 1：热路径零分配改造

**目标**：将 per-strategy 开销从 ~84ms 降到 ~5-8ms。
**工期**：3-5 天。**风险**：低。
**预期提升**：10-15x。

### 1.1 引入 LiveTradingObjectPoolManager

仿照回测模块 `BacktestObjectPoolManager`，新建实盘对象池。

**新增文件**：`SeverTest/Modules/StrategyEngine/Infrastructure/LiveTradingObjectPoolManager.cs`

```csharp
public sealed class LiveTradingObjectPoolManager
{
    private readonly SimpleObjectPool<StrategyExecutionContext> _contextPool;
    private readonly SimpleObjectPool<List<ConditionEvaluationResult>> _resultListPool;
    private readonly SimpleObjectPool<List<StrategyMethod>> _methodListPool;

    public LiveTradingObjectPoolManager()
    {
        _contextPool = new SimpleObjectPool<StrategyExecutionContext>(
            factory: () => new StrategyExecutionContext(),
            reset: ctx => ctx.Clear(),
            prewarm: 512);

        _resultListPool = new SimpleObjectPool<List<ConditionEvaluationResult>>(
            factory: () => new List<ConditionEvaluationResult>(32),
            reset: list => list.Clear(),
            prewarm: 1024);

        _methodListPool = new SimpleObjectPool<List<StrategyMethod>>(
            factory: () => new List<StrategyMethod>(16),
            reset: list => list.Clear(),
            prewarm: 2048);
    }

    public StrategyExecutionContext RentContext() => _contextPool.Rent();
    public void ReturnContext(StrategyExecutionContext ctx) => _contextPool.Return(ctx);
    public List<ConditionEvaluationResult> RentResultList() => _resultListPool.Rent();
    public void ReturnResultList(List<ConditionEvaluationResult> list) => _resultListPool.Return(list);
    public List<StrategyMethod> RentMethodList() => _methodListPool.Rent();
    public void ReturnMethodList(List<StrategyMethod> list) => _methodListPool.Return(list);
}
```

**注册**：`Program.cs` 添加 `services.AddSingleton<LiveTradingObjectPoolManager>();`

### 1.2 改造 ExecuteStrategyTask —— 消除逐策略分配

**当前代码**（`RealTimeStrategyEngine.cs:347-379`）：

```csharp
private void ExecuteStrategyTask(StrategyModel strategy, MarketDataTask task,
    StrategyRunMetrics metrics, StrategyRunTrace? trace)
{
    // ...
    var context = new StrategyExecutionContext(strategy, task, _valueResolver, _actionExecutor);
    var strategyStopwatch = Stopwatch.StartNew();
    // ...
}
```

**改为**：

```csharp
private void ExecuteStrategyTask(StrategyModel strategy, MarketDataTask task,
    StrategyRunMetrics metrics, StrategyRunTrace? trace)
{
    if (!IsRunnableState(strategy.State))
    {
        trace?.IncrementSkippedByState();
        Interlocked.Increment(ref metrics.SkippedCount);
        return;
    }

    trace?.IncrementRunnableStrategy();
    var context = _objectPool.RentContext();
    try
    {
        context.Reset(strategy, task, _valueResolver, _actionExecutor);
        var runtimeGate = ResolveRuntimeGate(strategy, context.CurrentTime);
        if (!runtimeGate.AllowEntry && !runtimeGate.AllowExit)
        {
            trace?.IncrementSkippedByRuntimeGate();
            Interlocked.Increment(ref metrics.SkippedCount);
            return;
        }

        ExecuteLogic(context, runtimeGate, metrics);
        metrics.AddExecutedStrategy(strategy.UidCode);
        Interlocked.Increment(ref metrics.ExecutedCount);
    }
    finally
    {
        _objectPool.ReturnContext(context);
    }
}
```

**变更要点**：
- `new StrategyExecutionContext` → `_objectPool.RentContext()` + `context.Reset()`
- 移除 `Stopwatch.StartNew()` —— 生产模式不需逐策略计时，用 trace 的整体 ExecuteMs 即可
- `StrategyExecutionContext` 已有 `Reset()` 和 `Clear()`（回测已实现），可直接复用

### 1.5 条件优先级预排序 —— 在 UpsertStrategy 时完成，不在每次 tick 重复

**当前问题**：`ConditionPriorityHelper.SplitByPriority` 在每次 `EvaluateChecks` 中、每个 group 都执行一次。
71 策略 × 4 groups = 284 次排序/tick。

**改法**：在 `UpsertStrategy` 时，对策略的每个 `ConditionGroup` 预排序：

```csharp
// 新增到 StrategyModel 或 ConditionGroup 上
public sealed class PreSortedConditionGroup
{
    public List<StrategyMethod> CrossRequired { get; init; }
    public List<StrategyMethod> CrossOptional { get; init; }
    public List<StrategyMethod> Required { get; init; }
    public List<StrategyMethod> Optional { get; init; }
    public int MinPassConditions { get; init; }
    public bool Enabled { get; init; }
}
```

在 `UpsertStrategy` 中调用一次 `SplitByPriority` 并缓存：

```csharp
public void UpsertStrategy(StrategyModel strategy)
{
    // ... 现有逻辑 ...
    PreSortConditions(strategy);
}

private void PreSortConditions(StrategyModel strategy)
{
    var logic = strategy.StrategyConfig?.Logic;
    if (logic == null) return;

    PreSortBranch(logic.Entry.Long);
    PreSortBranch(logic.Exit.Long);
}
```

这样 `EvaluateChecks` 就直接读取预排好序的列表，**完全跳过 SplitByPriority**，
也**不再需要 4 个临时 List 的分配和回收**。

### 1.6 移除热路径所有 LogDebug

**当前问题**：`EvaluateChecks` 每个条件都调用 `_logger.LogDebug(...)`，即使 Debug 级别未开启，
7 个参数的装箱操作仍然执行（C# 结构化日志的固有开销）。

**改法**：用编译时常量或 `_diagnostics` 选项守卫，完全跳过热路径日志：

```csharp
// 在 EvaluateCondition 局部函数中：
bool EvaluateCondition(StrategyMethod condition, bool isRequired)
{
    var result = _conditionEvaluator.Evaluate(context, condition);
    metrics?.IncrementConditionEval();
    results.Add(result);

    // 仅在诊断模式下才记录 Debug 日志
    // 正常执行路径完全跳过
    if (_diagnostics.LogEveryRunTask && debugEnabled)
    {
        _logger.LogDebug(...);
    }

    // ... 判断逻辑不变 ...
}
```

**同样处理以下位置**：
- `ExecuteBranch:525-530`（"策略检查失败"）
- `ExecuteBranch:536-540`（"策略检查通过"）
- `ExecuteBranch:554-558`（"策略检查通过，执行动作"）
- `ExecuteActions:808-815`（"动作执行"）
- `EvaluateEntryFilters:450-454`（"筛选器未通过"）

### 1.7 消除 HandleTask 中的冗余 Stopwatch

**当前**：`HandleTask` 创建了 5 个 Stopwatch（总、lookup、indicator、execute、per-strategy）。

**改为**：只保留一个总 Stopwatch + 用时间戳差值计算子阶段：

```csharp
private void HandleTask(MarketDataTask task)
{
    var startTicks = Stopwatch.GetTimestamp();
    var trace = _diagnostics.EnableRunProfileLog ? new StrategyRunTrace() : null;
    var metrics = new StrategyRunMetrics();
    // ...

    var lookupEndTicks = Stopwatch.GetTimestamp();
    // ... indicator ...
    var indicatorEndTicks = Stopwatch.GetTimestamp();
    // ... execute ...
    var executeEndTicks = Stopwatch.GetTimestamp();

    if (trace != null)
    {
        trace.LookupMs = TicksToMs(startTicks, lookupEndTicks);
        trace.IndicatorMs = TicksToMs(lookupEndTicks, indicatorEndTicks);
        trace.ExecuteMs = TicksToMs(indicatorEndTicks, executeEndTicks);
    }
}

private static int TicksToMs(long start, long end)
{
    return (int)((end - start) * 1000 / Stopwatch.Frequency);
}
```

### 1.8 消除 StrategyRunMetrics 中的 ConcurrentDictionary

**当前**：每次 `HandleTask` 创建 `StrategyRunMetrics`，其中包含 2 个 `ConcurrentDictionary`。
它们仅用于收集执行过的策略 ID 列表，供日志使用。

**改为**：用简单的 `List<string>` + 无竞争写（因为从 Parallel.ForEach 写入需锁），
或直接在诊断模式才收集：

```csharp
private sealed class StrategyRunMetrics
{
    // 基本计数器保留 Interlocked
    public int ExecutedCount;
    public int SkippedCount;
    public int ConditionEvalCount;
    public int ActionExecCount;
    public int OpenTaskCount;

    // 策略 ID 列表仅在诊断模式使用
    // 生产模式跳过收集
    private List<string>? _executedIds;
    private readonly object _idLock = new();

    public void AddExecutedStrategy(string uid, bool collectIds)
    {
        if (!collectIds) return;
        lock (_idLock)
        {
            _executedIds ??= new List<string>(64);
            _executedIds.Add(uid);
        }
    }
}
```

## Phase 2：指标异步预计算

**目标**：将 `UpdateIndicatorsBeforeExecute`（399ms）从 HandleTask 主路径移出。
**工期**：3-5 天。**风险**：中。
**预期提升**：indicator 阶段接近 0ms（读缓存）。

### 2.1 当前问题

```
HandleTask:
  1. UpdateIndicatorsBeforeExecute(399ms)  ← 同步阻塞
  2. Parallel.ForEach strategies(2990ms)
```

`UpdateIndicatorsBeforeExecute` 在策略执行前同步地：
1. 遍历所有匹配策略，收集需要的 IndicatorRequest
2. 调用 `_indicatorEngine.ProcessTaskNow(indicatorTask)` 同步计算
3. 每个指标走 `IndicatorHandle.Update()` → `TalibIndicatorCalculator.TryCompute()`

### 2.2 改造方案：行情驱动的实时指标更新

**核心思路**：IndicatorEngine 订阅 MarketDataEngine，行情一到就立即更新所有已注册指标。
当策略执行时，指标已是最新的，只需读缓存。

**改造步骤**：

**Step 1**：IndicatorEngine 新增 `SubscribeAndAutoUpdate` 方法：

```csharp
public async Task SubscribeAndAutoUpdateAsync(
    MarketDataEngine marketDataEngine, CancellationToken ct)
{
    var subscription = marketDataEngine.SubscribeMarketTasks(
        "IndicatorAutoUpdate", onlyBarClose: false);

    while (!ct.IsCancellationRequested)
    {
        var task = await subscription.ReadAsync(ct);
        // 直接处理所有已注册的指标
        var normalizedTask = NormalizeTask(task);
        foreach (var handle in _handles.Values)
        {
            if (handle.IsMatchingTask(normalizedTask))
            {
                handle.Update(normalizedTask, _marketDataProvider, _calculator, _logger);
            }
        }
    }
}
```

**Step 2**：`IndicatorHandle.IsMatchingTask` 需要改为 public。

**Step 3**：在 `StrategyRuntimeHostedService.ExecuteAsync` 中启动指标自动更新：

```csharp
var indicatorAutoTask = _indicatorEngine.SubscribeAndAutoUpdateAsync(
    _marketDataEngine, stoppingToken);

var workers = new[] { indicatorTask, strategyTask, indicatorAutoTask };
await Task.WhenAll(workers);
```

**Step 4**：`RealTimeStrategyEngine.HandleTask` 中的 `UpdateIndicatorsBeforeExecute` 改为只做验证：

```csharp
// 仅检查指标是否已就绪，不再同步计算
var indicatorReady = CheckIndicatorsReady(strategies.Values, task);
if (!indicatorReady)
{
    _logger.LogWarning("指标未就绪，跳过本次行情: {TraceId}", task.TraceId);
    return;
}
```

### 2.3 注意事项

- 指标自动更新和策略执行可能存在短暂的时间窗口（指标还在更新，策略已开始读）。
  由于 `IndicatorHandle.TryGetValue` 是无锁读取 `IndicatorSeries`，
  最多读到上一 tick 的值——对于 1m 级别行情，影响可忽略。
- 如果要保证严格一致性，可在 HandleTask 前加一个 `WaitForIndicatorFresh(task.CandleTimestamp)` 信号量。

### Phase 2 预期效果

HandleTask 主路径从 `399ms + 2990ms = 3390ms` 降为 `~0ms + 优化后的 execute`。

## Phase 3：外层并行 + 键分片

**目标**：从 1 worker 串行消费改为多 worker 并行消费，异键并行、同键串行。
**工期**：5-7 天。**风险**：中高。
**预期提升**：多热点键叠加场景线性加速。

### 3.1 当前问题

```
StrategyRuntimeHostedService:
  const int strategyWorkerCount = 1;  // 写死 1 个 worker
```

所有 MarketDataTask 串行消费：BTC/USDT 跑完才能跑 ETH/USDT。

### 3.2 改造方案：KeyShardedConsumer

**核心设计**：按 `exchange|symbol|timeframe` 分片，每个分片独立消费。

```
MarketDataTask 到达
    ↓
KeyRouter 按 key hash 路由到 shard
    ↓
Shard[0]: BTC/USDT 1m  → 串行处理该 key 的所有 task
Shard[1]: ETH/USDT 1m  → 串行处理该 key 的所有 task
Shard[2]: BTC/USDT 15m → 串行处理该 key 的所有 task
...
```

**新增类**：`KeyShardedStrategyConsumer`

```csharp
public sealed class KeyShardedStrategyConsumer
{
    private readonly int _shardCount;
    private readonly Channel<MarketDataTask>[] _shardChannels;
    private readonly RealTimeStrategyEngine _engine;
    private readonly SemaphoreSlim _globalParallelismBudget;

    public KeyShardedStrategyConsumer(
        RealTimeStrategyEngine engine,
        int shardCount,
        int maxGlobalParallelism)
    {
        _engine = engine;
        _shardCount = shardCount;
        _shardChannels = new Channel<MarketDataTask>[shardCount];
        _globalParallelismBudget = new SemaphoreSlim(maxGlobalParallelism);

        for (var i = 0; i < shardCount; i++)
        {
            _shardChannels[i] = Channel.CreateBounded<MarketDataTask>(
                new BoundedChannelOptions(256)
                {
                    SingleReader = true,
                    SingleWriter = false,
                    FullMode = BoundedChannelFullMode.DropOldest
                });
        }
    }

    public void Route(MarketDataTask task)
    {
        var key = $"{task.Exchange}|{task.Symbol}|{task.TimeframeSec}";
        var shard = Math.Abs(key.GetHashCode()) % _shardCount;
        _shardChannels[shard].Writer.TryWrite(task);
    }

    public Task RunAsync(CancellationToken ct)
    {
        var workers = new Task[_shardCount];
        for (var i = 0; i < _shardCount; i++)
        {
            var shardIndex = i;
            workers[i] = Task.Run(() => ConsumeShardAsync(shardIndex, ct), ct);
        }
        return Task.WhenAll(workers);
    }

    private async Task ConsumeShardAsync(int shardIndex, CancellationToken ct)
    {
        var reader = _shardChannels[shardIndex].Reader;
        while (await reader.WaitToReadAsync(ct))
        {
            while (reader.TryRead(out var task))
            {
                // 同一 shard 内串行，保证同 key 时序
                _engine.HandleTask(task);
            }
        }
    }
}
```

### 3.3 全局并行度预算

关键：**内层 Parallel.ForEach 的并行度必须与外层 shard 数协调**。

```
总线程预算 = Environment.ProcessorCount（例如 16）
shard 数 = 4（配置化）
每 shard 内层并行度 = 16 / 4 = 4

// 或者更智能：用 SemaphoreSlim 做全局预算
每 shard 的 Parallel.ForEach 前先 Acquire N 个许可，执行完释放
```

在 `RealTimeStrategyEngine` 中改造：

```csharp
// 构造时注入全局并行度预算
private readonly SemaphoreSlim? _globalParallelismBudget;

// HandleTask 中：
if (_globalParallelismBudget != null)
{
    // 动态计算本次需要的并行度
    var requestedParallelism = Math.Min(matchedCount, _maxParallelism);
    var granted = TryAcquireParallelism(requestedParallelism);
    _parallelOptions.MaxDegreeOfParallelism = granted;
}
```

## Phase 4：TalibWasm 并行化

**目标**：解除 `TalibWasmNodeInvoker` 的全局 `lock (_sync)` 瓶颈。
**工期**：3-5 天。**风险**：中。
**预期提升**：indicator 阶段（Phase 2 前） 2-4x。

### 4.1 当前问题

```csharp
// TalibWasmNodeInvoker.cs:60
lock (_sync)
{
    if (!EnsureProcessStarted(out error))
    {
        return false;
    }
    // ... JSON 序列化 → 写 stdin → 读 stdout → JSON 反序列化 ...
}
```

所有 TALib 计算串行通过同一个 Node.js 子进程的 stdin/stdout。
即使 IndicatorEngine 有多个 worker，它们在 `lock (_sync)` 上排队。

### 4.2 改造方案 A：多进程池

```csharp
public sealed class TalibWasmNodePool : IDisposable
{
    private readonly TalibWasmNodeInvoker[] _invokers;
    private int _roundRobin;

    public TalibWasmNodePool(int poolSize, IOptions<TalibCoreOptions> options,
        ILoggerFactory loggerFactory)
    {
        _invokers = new TalibWasmNodeInvoker[poolSize];
        for (var i = 0; i < poolSize; i++)
        {
            _invokers[i] = new TalibWasmNodeInvoker(
                options, loggerFactory.CreateLogger<TalibWasmNodeInvoker>());
        }
    }

    public TalibWasmNodeInvoker Acquire()
    {
        var index = Interlocked.Increment(ref _roundRobin) % _invokers.Length;
        return _invokers[Math.Abs(index)];
    }
}
```

配置：`TalibCore.PoolSize: 4`（默认 = ProcessorCount / 4）

### 4.3 改造方案 B（长期）：原生 .NET TALib

用 TA-Lib .NET 原生绑定（如 `TALib.NETCore` NuGet 包），
彻底消除 IPC + JSON 序列化开销。

```csharp
// 替换 TalibWasmNodeInvoker.TryCompute 为直接调用：
var result = TALib.Core.Sma(input, 0, input.Length - 1,
    output, out var outBeg, out var outCount, period);
```

**优势**：
- 零 IPC 开销
- 零 JSON 开销
- 线程安全（TALib 函数无状态）
- 可完全并行

**风险**：需要验证计算结果与 WASM 版本完全一致（同源化）。

  ├─ 2.1-2.2 IndicatorEngine 订阅行情自动更新
  ├─ 2.3 HandleTask 移除同步指标计算
  └─ 压测验证：HandleTask indicator 阶段 < 5ms

Week 3-4: Phase 3（外层并行 + 键分片）
  ├─ 3.1-3.2 KeyShardedStrategyConsumer
  ├─ 3.3 全局并行度预算
  └─ 压测验证：4 热点键并行 < 3s 总耗时

Week 4-5: Phase 4（TalibWasm 并行化）
  ├─ 4.1-4.2 多进程池
  └─ （可选）原生 TALib 迁移

验收标准（3秒安全水位）：
  - 单热点键 1000 策略：P95 < 2s
  - 4 热点键各 500 策略：P95 < 3s
  - 跨分钟堆积率 < 0.01%
```

## 风险与回退

| 阶段 | 主要风险 | 回退方案 |
|------|---------|---------|
| Phase 1 | 对象池回收遗漏导致内存泄漏 | 添加 pool 监控指标；发现泄漏可 disable pool 回退到 new |
| Phase 2 | 指标未就绪时策略读到旧值 | 保留 `UpdateIndicatorsBeforeExecute` 作为 fallback，配置开关切换 |
| Phase 3 | 分片不均导致某 shard 过热 | 动态 rebalance；或回退到 1 worker |
| Phase 4 | 多 Node 进程资源占用过高 | 减少 pool size；或回退到单进程 |

```

  ├─ [IndicatorAutoUpdate 通道] → IndicatorEngine.PrecomputeHandlesForTask()

**兼容性**：

- 预计算 worker 与 HandleTask 可能同时更新同一 handle，lock 保证串行，freshness check 避免重复计算

### 2026-02-26 Phase 3 落地（外层并行 + 键分片）

**变更文件**：

1. **新增** `SeverTest/Modules/StrategyRuntime/Application/KeyShardedStrategyConsumer.cs`
   - 按 `exchange|symbol|timeframe` 哈希路由到 N 个分片 Channel
   - 每个分片单 reader 消费（保证同键时序），异键并行

   - 新增 `internal AdjustInnerParallelism(int shardCount)`：按分片数调低内层 `Parallel.ForEach` 并行度（`ProcessorCount / shardCount`）

   - 构造函数新增 `IOptions<LiveTradingOptions>?` 可选参数
   - `ExecuteAsync`：当 `ShardCount > 1` 时创建 `KeyShardedStrategyConsumer` 替代 `RunWorkersAsync(1)`

   - 新增 `"LiveTrading": { "ShardCount": 1 }` 配置段

6. **配置注册** `SeverTest/Program.cs`
   - 新增 `services.Configure<LiveTradingOptions>(configuration.GetSection("LiveTrading"))`

**并行度协调**：
- 外层 shardCount 个 worker 并行消费不同键
- 内层 `Parallel.ForEach` 并行度自动调整为 `ProcessorCount / shardCount`（最低 2）
- 总线程使用量 ≈ `shardCount × (ProcessorCount / shardCount)` = `ProcessorCount`，避免线程爆炸

**兼容性**：
- `ShardCount = 1`（默认）时走原有单 worker 路径，零行为变更
- `LiveTradingOptions` 为可选注入，未配置时自动 fallback 到默认值
- `ProcessTask` 和 `AdjustInnerParallelism` 为 `internal`，不影响公开 API
