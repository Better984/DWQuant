# 回测系统需求说明

## 0. 最终成品效果与端到端链路

### 0.1 用户侧体验
- 用户在前端**选择多个策略**（例如 3 个），可不同周期、不同币种。
- 填写回测参数（时间范围/根数、初始资金、手续费、滑点等）。
- 一键提交，发送给后端回测服务器。
- 通过 WebSocket 或轮询获取进度，任务完成后查看**一根合并资金曲线**及统计指标。

### 0.2 后端端到端链路（确保正确、链路流通）

```
用户选择多策略 → 提交请求
        ↓
┌─────────────────────────────────────────────────────────────────────┐
│ 1. 获取 K 线                                                         │
│    按各策略的 exchange/symbol/timeframe 拉取历史 K 线，构建统一时间轴  │
└─────────────────────────────────────────────────────────────────────┘
        ↓
┌─────────────────────────────────────────────────────────────────────┐
│ 2. 计算指标                                                          │
│    对每个策略、每个标的，按条件配置计算所需指标（MA、RSI 等）          │
└─────────────────────────────────────────────────────────────────────┘
        ↓
┌─────────────────────────────────────────────────────────────────────┐
│ 3. 统计满足条件的位置                                                 │
│    遍历时间轴，评估开仓条件，记录满足条件的检查点（候选开仓）          │
└─────────────────────────────────────────────────────────────────────┘
        ↓
┌─────────────────────────────────────────────────────────────────────┐
│ 4. 筛选仓位                                                           │
│    按 MaxPositionQty、运行时间门禁等风控规则过滤，得到有效开仓        │
└─────────────────────────────────────────────────────────────────────┘
        ↓
┌─────────────────────────────────────────────────────────────────────┐
│ 5. 平仓模拟                                                           │
│    对有效开仓逐根 K 线检测止盈/止损触发，生成完整交易明细              │
└─────────────────────────────────────────────────────────────────────┘
        ↓
┌─────────────────────────────────────────────────────────────────────┐
│ 6. 统计计算                                                           │
│    基于合并后的交易列表，计算资金曲线、夏普、Sortino、最大回撤等       │
└─────────────────────────────────────────────────────────────────────┘
        ↓
返回结果（合并资金曲线、交易明细、统计指标）
```

### 0.3 正确性保障
- **时序一致**：多策略、多周期按统一时间轴推进，避免跨时间乱序。
- **资金一致**：多策略共享同一资金池，权益累计与实盘逻辑一致。
- **风控一致**：MaxPositionQty、运行时间门禁等与实盘执行逻辑对齐。
- **结果可复现**：相同输入（策略、参数、K 线）得到相同输出。

### 0.4 与当前实现的关系
- **单策略多标的**：当前已支持，链路完整。
- **多策略组合**：当前已支持 P1（策略腿执行 + 结果合并）；P2 可选增强为统一时间轴组合执行，见第 6 节。

## 1. 产品定位与场景

### 1.1 产品类型
- **C 端量化产品**：面向普通用户，支持多用户同时使用。
- **多租户**：多个用户可独立创建、提交、查看各自的回测任务。
- **高并发**：需支持多用户同时提交回测，任务排队与限流。

### 1.2 典型使用场景
- 用户创建策略后，在策略详情页发起回测。
- 用户填写回测参数（交易所、标的、周期、时间范围/根数、初始资金、手续费、滑点等）。
- 任务进入队列，用户通过 WebSocket 或 HTTP 轮询获取进度。
- 任务完成后查看交易明细、资金曲线、统计指标（含夏普、Sortino、Calmar 等）。

---

## 2. 任务队列与生命周期

### 2.1 任务状态流转

```
queued（排队中） → running（执行中） → completed（已完成）
                                    → failed（失败）
       → cancelled（已取消）
```

### 2.2 任务等待队列
- 用户提交回测后，任务进入 `backtest_task` 表，状态为 `queued`。
- 本地 Worker 或远端算力节点轮询队列，抢占任务后置为 `running`。
- 队列容量由 `Backtest:MaxQueueSize` 控制，超出时拒绝提交并提示“回测队列已满”。
- 单用户并发由 `Backtest:MaxTasksPerUser` 控制，超出时提示“当前有 N 个回测任务进行中，单用户最多 M 个”。

### 2.3 任务取消
- **未开始的任务可被取消**：仅 `queued` 状态的任务允许用户取消。
- `running` 状态的任务原则上不提供取消（避免执行中途中断导致结果不一致；若需支持，需在实现中增加 CancellationToken 传递与优雅退出）。
- 取消后任务状态置为 `cancelled`，`completed_at` 记录取消时间。

## 3. 性能要求

### 3.1 极高性能目标
- 回测为计算密集型任务，需充分利用多核 CPU。
- 单任务内部：开仓检测、平仓模拟、指标计算等环节应支持并行。
- 多任务：支持多任务并发执行，由 `Backtest:MaxConcurrentTasks` 控制。

### 3.2 CPU 核心数获取
- **启动时尽可能获取 CPU 核心数量**：使用 `Environment.ProcessorCount` 作为并行度基准。
- 以下配置项在 `<=0` 时按 CPU 核心数生效：
  - `Backtest:InnerParallelism`：单任务主循环 symbol 并行度。
  - `Backtest:BatchCloseParallelism`：批量模式平仓并行度。
- 建议：启动时日志输出检测到的 CPU 核心数，便于运维确认并行配置是否合理。

## 4. 批量执行模式（batch_open_close）

### 4.1 执行流程概览

```
第一阶段：统一检测开仓
    ↓
第二阶段：统一开仓处理（含过滤）
    ↓
第三阶段：平仓模拟（止盈/止损/强平）
    ↓
第四阶段：资金曲线与统计指标计算
```

### 4.2 第一阶段：统一检测开仓
- 遍历所有标的 × 驱动时间轴，逐检查点评估条件是否满足开仓。
- 满足条件的检查点记为 `BatchOpenCandidate`（候选开仓）。
- 并行执行：按 symbol 并行，每个 symbol 内按时间轴串行。
- 输出：每个标的的候选开仓列表。

### 4.3 第二阶段：统一开仓处理 + 过滤
- **开仓处理**：对候选开仓按时间顺序执行“开仓”逻辑（记录成交价、数量、手续费等）。
- **过滤处理**：回测必须真实模拟实盘风控规则，包括但不限于：
  - **最大持仓限制（MaxPositionQty）**：策略配置 `trade.sizing.maxPositionQty` 时，同向持仓累计不得超过该值。若 `当前同向持仓 + 本次开仓量 > maxPositionQty`，则**阻断**本次开仓，不纳入最终交易。
  - 其他策略级风控（如运行时间门禁等）在开仓检测阶段已生效。
- 过滤后的仓位才是“有效开仓”，后续平仓与统计仅基于有效开仓。

### 4.4 第三阶段：平仓模拟
- 对有效开仓，按 K 线时间戳顺序逐根检测止盈/止损是否触发。
- 触发则记录平仓时间、价格、PnL、手续费等。
- 支持并行：按仓位分片，多线程并行模拟平仓。
- 非重叠模式：同一标的同时仅允许一个方向的持仓，需按开仓时间正序处理，避免重叠。

### 4.5 第四阶段：资金曲线与统计指标
- **资金曲线**：基于过滤后的交易列表，按时间顺序累计权益，并按粒度聚合（1m/15m/1h/4h/1d/3d/7d）。
- **统计指标**：
  - 基础：总收益、总收益率、最大回撤、胜率、交易次数、平均盈亏、盈亏比等。
  - 高级：夏普比率、Sortino 比率、年化收益率、Calmar 比率、最大连续亏损/盈利、平均持仓时间、最大回撤持续时间等。

## 5. 当前实现与阶段计划

### 5.1 已实现
- 任务队列与排队机制。
- 批量模式：统一开仓检测 + 并行平仓。
- 资金曲线与统计指标计算。
- 支持多标的、多 symbol 并行。
- 取消任务（当前仅支持取消 queued）。
- 并行度配置：`InnerParallelism`、`BatchCloseParallelism` 在 `<=0` 时按 CPU 核心数。
- 超级测试台回测压测入口：支持批量创建随机策略并并发提交回测任务，返回提交耗时分位、状态分布与线程/进程快照，便于阶段化性能验收。

### 5.2 阶段 P1（已完成，可交互验收）
- **MaxPositionQty 过滤**：已在批量模式落地。按开仓时间顺序维护“在途同向持仓量”，当 `当前在途持仓 + 本次候选开仓 > maxPositionQty` 时阻断候选；阻断数量会通过 `blockedPositions` 推送并写入阶段日志。
- **取消仅限 queued**：`CancelAsync` 已收敛为仅允许 `status = 'queued'`。
- **启动时 CPU 检测日志**：`BacktestTaskWorker` 启动时输出 CPU 核心数、任务并发上限、单任务并行度。
- **多策略组合回测（P1 口径）**：已启用 `strategies` 请求路径，支持按策略腿执行并合并结果；输出组合总资金曲线 `totalEquityCurveRaw`、`totalEquitySummary`，并在交易/标的结果透传 `strategyScopeKey`、`strategyIndex` 便于定位来源策略。

### 5.3 阶段 P2（可选增强，非阻塞当前上线）
- **组合执行模型升级**：从“策略腿串行执行 + 结果合并”升级为“组合统一时间轴推进（事件驱动）”。
- **执行层资金约束**：在开仓执行时引入组合账户可用资金/保证金约束，做到“先约束、后成交”。
- **组合任务内并行优化**：策略腿与指标计算进一步并行化，并控制 CPU 争抢，提升单任务峰值吞吐。

## 6. 多策略组合回测（阶段化）

### 6.1 需求描述
- 支持**一次性回测多个策略**：例如同时回测 3 个策略，每个策略可不同周期、不同币种。
- 三个策略共享同一资金池，输出**一根由三者共同影响的合并资金曲线**。
- 统计指标基于合并后的交易与权益计算。

### 6.2 当前落地（P1）
- 当前：支持“单策略”与“多策略组合”双路径。
- 组合路径：按策略腿执行并合并为同一资金池口径的总权益曲线（串行执行，结果合并）。

### 6.3 实现要点（P1）
- 请求层：支持 `strategies: [{ usId/configJson, symbols, timeframe, strategyScopeKey }, ...]`。
- 执行层：`BacktestService` 构建策略腿执行计划，逐腿调用 `BacktestRunner`，复用现有高性能执行链路。
- 资金模型：按“共享单份初始资金 + 各策略净PnL叠加”构建组合总资金曲线。
- 输出层：返回组合 `totalEquityCurveRaw`、`totalEquitySummary`、`totalStats`，并在 `symbols/trades` 保留策略腿标识。
- 进度链路：新增 `portfolio_prepare` / `portfolio_run` / `portfolio_merge` 阶段推送。

### 6.4 可选增强（P2）
- 统一时间轴组合执行：多策略在同一驱动时间轴下逐点评估与执行。
- 执行层共享资金约束：将可用资金/保证金约束前置到开仓执行阶段。
- 组合级风险联动：支持组合层风控（如组合总仓位、组合最大回撤阈值）。

## 7. 配置项汇总

| 配置键 | 类型 | 说明 | 默认 |
|--------|------|------|------|
| `Backtest:MaxConcurrentTasks` | int | 全局最大并发回测任务数 | 3 |
| `Backtest:InnerParallelism` | int | 单任务 symbol 并行度（`<=0` 按 CPU 核心数） | 4 |
| `Backtest:BatchCloseParallelism` | int | 批量模式平仓并行度（`<=0` 按 CPU 核心数） | 0 |
| `Backtest:MaxTasksPerUser` | int | 单用户最大并发任务数 | 2 |
| `Backtest:MaxQueueSize` | int | 队列最大排队数量 | 20 |
| `Backtest:MaxBarCount` | int | 单次回测最大 K 线数 | 500000 |
| `Backtest:TaskTimeoutMinutes` | int | 单任务最大执行时间（分钟） | 10 |
| `Backtest:ExecutionModeDefault` | string | 默认执行模式（`batch_open_close`/`timeline`） | batch_open_close |
| `Backtest:BatchAllowOverlappingPositions` | bool | 批量模式是否允许重叠仓位 | false |
| `Backtest:WorkerPollingIntervalMs` | int | 任务轮询间隔（毫秒） | 500 |

## 8. 可行性评估（基于当前代码）

### 8.1 与实现一致的部分
| 需求项 | 代码现状 | 结论 |
|--------|----------|------|
| 任务队列与排队 | `BacktestTaskService.SubmitAsync` 限流，`TryAcquireQueuedTaskAsync` 原子抢占 | ✓ 已实现 |
| 批量模式四阶段 | 开仓检测 → 开仓过滤 → 平仓模拟 → 权益构建 | ✓ 已实现 |
| 资金曲线与统计 | `BuildBatchEquityCurve`、`BacktestStatisticsBuilder` | ✓ 已实现 |
| 并行度按 CPU | `ResolveInnerParallelism`、`ResolveBatchCloseParallelism` 在 `<=0` 时用 `Environment.ProcessorCount` | ✓ 已实现 |
| 重叠过滤 | `allowOverlappingPositions=false` 时在 `CloseCandidatesForSymbol` 中按开仓时间正序过滤 | ✓ 已实现 |
| 多策略组合回测（P1） | `BacktestService` 启用组合执行与合并输出 | ✓ 已实现 |

### 8.2 需调整或补充的部分
| 需求项 | 代码现状 | 可行性 |
|--------|----------|--------|
| **取消仅限 queued** | 已调整为仅允许取消 `queued` | 已完成 |
| **MaxPositionQty 过滤** | 已在批量模式落地，按时间顺序累计在途持仓并阻断超限候选 | 已完成 |
| **启动时 CPU 日志** | `BacktestTaskWorker` 启动日志已输出核心数与并行配置 | 已完成 |
| **多策略组合回测（P1）** | `strategies` 已可执行，返回组合资金曲线与组合统计 | 已完成 |
| **统一时间轴组合执行（P2）** | 当前为策略腿串行执行后合并 | 可增强 |
| **执行层资金约束（P2）** | 当前组合资金约束主要在结果合并口径体现 | 可增强 |

### 8.3 需澄清的语义
- **MaxQueueSize**：当前 `CountActiveGlobalAsync` 统计的是 `queued + running` 总数，即“活跃任务数”上限，而非仅排队数。文档中的“队列容量”可理解为“系统可承载的活跃任务数”，与实现一致。

### 8.4 结论
**需求文档可行，且当前版本已落地 P1 核心能力**。当前实现可直接用于 C 端多用户回测与多策略组合交互验收；P2 项属于“性能与语义增强”，不阻塞当前上线压测。
