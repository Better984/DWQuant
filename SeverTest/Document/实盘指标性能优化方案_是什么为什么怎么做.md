# 实盘指标性能优化方案（是什么 / 为什么 / 怎么做）

> 目的：统一后续实盘指标链路优化方向，先解决高收益热点，确保“策略数量上来后仍能稳定、快速执行”。

---

## 1. 是什么（当前瓶颈是什么）

### 1.1 当前链路概览

实盘链路中，指标相关核心路径为：

1. `MarketDataEngine` 产出 `MarketDataTask`
2. `IndicatorEngine` 预计算 + `RealTimeStrategyEngine` 执行前刷新
3. `IndicatorValueResolver` / `ConditionMethodRegistry` 取值并评估条件

相关代码位置：

- `SeverTest/Modules/MarketStreaming/Application/MarketDataEngine.cs`
- `SeverTest/Modules/StrategyEngine/Application/IndicatorEngine.cs`
- `SeverTest/Modules/StrategyEngine/Application/RealTimeStrategyEngine.cs`
- `SeverTest/Modules/StrategyEngine/Application/IndicatorValueResolver.cs`
- `SeverTest/Modules/StrategyEngine/Application/TalibIndicatorCalculator.cs`

### 1.2 已确认性能热点（当前确实存在）

1. `GetHistoryKlines(string, ...)` 每次调用都会“复制+过滤+切片”，在高频场景下产生大量无效扫描与分配。

2. 同一条 `MarketDataTask` 内，字段解析与指标解析会重复拉取高度重叠的 K 线片段，缺少任务级复用缓存。

3. `IndicatorEngine` 预计算阶段通过 `foreach (_handles.Values)` 扫描全部句柄，句柄越多，开销线性放大；且当前缺少明确的句柄回收机制。

4. `OnBarUpdate` 任务会进入策略执行主流程，但很多策略实际上只依赖 `OnBarClose`，存在可提前短路的无效执行。

5. 条件评估前每次都动态构建 condition key 并做 MD5，条件数量上来后这部分 CPU/分配成本明显。

6. 条件方法中大量 `StringBuilder` 消息构建与部分 `Info` 级日志处于热路径，会放大 GC 和日志 I/O 压力。

7. TA-Lib 计算外围仍有较多输入/输出数组分配；在 K 线切片未优化时会被进一步放大。

---

## 2. 为什么（为什么会慢，为什么要按这个顺序做）

### 2.1 根因判断

当前慢点主要在“指标外围数据准备与重复计算”，而不是 TA-Lib 核心算子本身：

1. 高频的 K 线全表处理（复制/LINQ）导致 CPU 和内存分配持续放大。
2. 同任务内同市场数据重复读取，导致重复 I/O 与重复对象创建。
3. 句柄全表扫描和无效策略执行使计算复杂度随策略规模线性恶化。

### 2.2 为什么优先改 K 线切片与复用

1. 改动局部、风险低：集中在数据访问层和执行上下文缓存，不改业务语义。
2. 收益直接：减少全表扫描、降低分配次数，通常能立刻改善耗时与 GC。
3. 能为后续优化打基础：先把输入阶段收敛，再考虑增量指标核、缓冲池化等深层优化。

## 3. 怎么做（分阶段落地）

## Phase 0：基线与观测（先做）

目标：先建立可对比基线，避免“改了但无证据”。

执行项：

1. 固定压测样本：策略数、每策略条件数、时间周期组合固定。
2. 记录基线指标：
   - 单任务总耗时 P50/P95/P99
   - 指标阶段耗时（`IndicatorMs`）
   - 每秒任务处理数（TPS）
   - GC 指标（Gen0/Gen1 次数、分配速率）
3. 统计 `GetHistoryKlines` 调用次数与平均返回长度。

验收输出：形成一份“优化前基线”快照（日志或表格）。

## Phase 1：高收益低风险改造（优先级 P0）

### 3.1 重写 `GetHistoryKlines(string, ...)`（去复制/去 LINQ）

目标：把“全表复制+过滤+切片”改为“索引定位+窗口拷贝”。

做法：

1. 在锁内直接基于 `candles` 定位 `lastIndex`：
   - 无 `endTimestamp`：`lastIndex = candles.Count - 1`
   - 有 `endTimestamp`：二分找到 `timestamp <= endTimestamp` 的最后下标
2. 计算 `take = min(count, lastIndex + 1)`，再用 `for` 循环拷贝窗口。
3. 非 `1m` 且存在 `currentBucket` 时按规则决定是否补入（保持原业务语义）。

预期收益：

1. 从“全表多次遍历”降到“按需窗口遍历”。
2. 显著降低临时列表与 LINQ 枚举器分配。

### 3.2 增加任务级 K 线本地缓存（字段/指标共享）

目标：同一 `MarketDataTask` 内，不重复拉取同一市场片段。

做法：

1. 在执行上下文增加轻量缓存：
   - key 建议：`(exchange, symbol, timeframe, endTimestamp, needCountBucket)`
2. `IndicatorValueResolver.TryResolveField` 先查本地缓存，未命中再走 provider。
3. `IndicatorEngine` 在同一任务处理内复用同一 `(exchange,symbol,timeframe)` 的 K 线窗口。

预期收益：

1. 把同任务内 K 线读取次数从“字段数+指标数级别”压到“少量唯一 key 级别”。
2. 降低锁竞争和重复分配。

---

## Phase 2：结构性优化（优先级 P1）

### 3.3 句柄分桶索引 + 回收机制

目标：避免 `IndicatorEngine` 每次全量扫 `_handles`。

做法：

1. 建立二级索引：`exchange|symbol|timeframe -> handle 列表`。
2. 预计算仅扫描命中分桶。
3. 引入引用计数：策略 `Upsert` 增、`Remove` 减，归零即移除句柄。

预期收益：

1. 预计算复杂度由 O(全句柄) 收敛到 O(分桶命中句柄)。
2. 避免句柄长期膨胀造成性能劣化。

### 3.4 OnBarUpdate 快速短路

目标：让不依赖更新态的策略不进入完整执行流程。

做法：

1. 策略注册时预编译标记：是否存在 `OnBarUpdate` 依赖。
2. `task.IsBarClose == false` 且策略无 update 依赖时直接跳过。

预期收益：减少无效条件评估与方法调用。

---

## Phase 3：热路径精修（优先级 P2）

### 3.5 条件 key 预编译

目标：减少运行期 `BuildKey + MD5` 开销。

做法：在 `UpsertStrategy` 时预生成每个条件的 key，运行期直接取 key。

### 3.6 条件消息惰性构建

目标：减少 `StringBuilder` 分配。

做法：默认仅返回布尔结果与简短码；详细消息仅在 debug/慢日志模式构建。

### 3.7 日志降级与限流

目标：避免热路径日志 I/O 放大。

做法：

1. 高频软失败日志降为 `Debug`。
2. 按 key 做采样（如 60 秒内同 key 最多 N 条）。
3. 移除热路径 `Console.WriteLine`，统一走 `ILogger`。

## 5. 验收标准（必须量化）

每阶段完成后都要对比 Phase 0 基线，至少满足：

1. `IndicatorMs` P95 明显下降（目标 ≥ 30%）。
2. 单任务总耗时 P95 下降（目标 ≥ 20%）。
3. 分配速率下降、GC 频率下降（Gen0/Gen1 次数下降趋势明确）。
4. 功能语义不变：
   - 条件命中结果一致
   - 开平仓动作一致
   - `OnBarClose/OnBarUpdate` 行为一致

## 6. 风险与回滚

1. 风险：K 线切片边界（`endTimestamp`、bucket 合并）处理不当会导致取值偏移错误。  
   控制：增加边界单测（空数据、极小 count、刚好边界、跨周期）。

2. 风险：任务级缓存 key 设计不当可能引入脏读。  
   控制：缓存生命周期限定在单任务/单执行上下文，任务结束即释放。

3. 风险：OnBarUpdate 短路误判导致策略漏执行。  
   控制：注册期静态分析 + 运行期保护开关，可快速回退。

回滚策略：

1. 每个阶段独立开关（配置项控制）。
2. 发现异常时仅回退当前阶段，不影响已验证阶段。

### 8.4 2026-02-27 补充修复（正确性热修）

1. 修复“误判已预计算”风险（高优先级）：  
问题：

### 8.6 2026-02-27 回归复盘与性能修复（历史复盘，已由极简链路替代）

1. 回归现象（基于运行日志）：

   - `SeverTest/Modules/StrategyEngine/Application/IndicatorHandle.cs`
