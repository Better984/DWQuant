# TA指标同源化方案（概要版）

## 1. 背景与前因后果

当前 `DWQuant` 的指标计算已经分布在两端：

- 前端 `Client` 使用 `talib-web + talib.wasm` 本地计算。
- 后端 `SeverTest` 使用 `TALib.NETCore` 计算，用于策略引擎、回测和运行时判定。

业务目标很明确：

1. 前端必须可以独立计算，不依赖后端指标接口，避免集中式计算带来的服务器压力。
2. 后端仍然需要指标计算能力（策略执行、回测、风控都要用）。
3. 两端计算结果要尽可能做到“完全一致”，避免出现“图上看到一个值、策略用另一个值”。

这本质上是一个“同源一致性”问题，不是单纯的性能问题。

## 2. 为什么要这么做

### 2.1 业务与产品层面

- **降低服务器成本与峰值压力**：指标计算从后端下沉到浏览器，服务器不再承担高频可视化指标请求。
- **提升用户体验**：前端本地更新指标，交互延迟更低。
- **避免信任危机**：交易产品最忌讳“展示值”和“策略值”不一致。

### 2.2 技术层面

- 目前两端都已接入 TA-Lib 生态，具备同源化基础。
- 主要风险不在“有没有库”，而在“同一个指标的输入预处理、参数取整、空值处理、输出对齐是否一致”。

## 3. 目标与范围

### 3.1 目标

- 前端 React 独立完成指标计算，不依赖 C# 计算接口。
- 后端 C# 保留指标计算能力。
- 两端统一规则，达到可验证的一致性（定义误差阈值与对齐测试）。

### 3.2 非目标

- 不把所有策略逻辑搬到前端。
- 不要求前后端完全共用二进制运行时（允许不同语言运行时，但必须同规则同数据同参数）。

## 4. 总体设计（概要）

采用“**同源规范层 + 双端执行适配层**”方案：

1. **同源规范层（唯一真源）**
   - 统一指标清单：指标代码、输入定义、参数类型、输出定义。
   - 统一计算规则：输入映射、数值预处理、参数取整、NaN 处理、输出对齐。

2. **前端执行层（React）**
   - 使用 `talib-web + wasm`。
   - 在 `Web Worker` 中执行指标，主线程只渲染。
   - 首次访问自动下载 wasm，用户不需要额外安装或编译。

3. **后端执行层（C#）**
   - 使用 `TALib.NETCore`。
   - 与前端共享同一份规范层配置与规则实现。

## 5. 一致性保障策略

- 建立“黄金样本”测试：同一组 OHLCV + 参数，在前端与后端分别计算并逐点比对。
- 统一误差标准：浮点结果按 `abs(a-b) <= 1e-10`（可按指标组调整）。
- 统一边界行为：空值、窗口不足、整数参数取整、输出索引映射。

## 6. 实施阶段

### 阶段A：规则对齐

- 固化统一规范文件。
- 修正前后端差异点（尤其是空值与取整规则）。

### 阶段B：一致性测试

- 全量指标自动化对齐测试。
- 输出差异报告并修正。

### 阶段C：上线与运维

- 前端开启 Worker 计算与缓存。
- 后端仅保留策略/回测计算，不对外承担图表指标计算压力。

## 7. 预期收益

- 服务器压力显著下降。
- 客户端指标响应速度提升。
- 指标展示与策略执行的数值一致性可证明、可回归、可审计。

## 8. 结论

该方案在当前 `DWQuant` 代码基础上可直接落地，属于“中风险、可控改造”。
核心成功条件是：**统一规则并建立自动化对齐测试**，而不是继续各端独立“各算各的”。

## 10. 新增随机一致性测试工具（2026-02-21）

### 已实现

1. 后端新增协议接口：`marketdata.ta.random.compare`（`POST /api/MarketData/ta-random-compare`）。
2. 接口行为：
   - 从历史缓存快照中随机选择交易所/币种/周期组合；
   - 随机截取连续 `2000` 根 K 线窗口；
   - 对当前支持的 TA 指标逐一调用同核心（`Node + talib-web + talib.wasm`）计算；
   - 返回样本元信息、K 线序列、每个指标的输入源/参数/输出数组。
3. 前端 `Dashboard` 测试区新增入口：`随机一致性测试`。
4. 前端弹窗新增能力：
   - 一键触发随机测试；
   - 对同一组 K 线在前端本地 `talib-web` 重算全部指标；
   - 按“指标-输出”维度逐项对比并展示：最大误差、首差异索引、通过/失败；
   - 支持查看单个输出的全量逐根明细（后端值 / 前端值 / 绝对误差）。

### 价值

1. 可直接在产品界面做“端到端数值一致性回归”，无需人工导出脚本。
2. 能快速发现输入映射、参数处理、输出对齐等细节偏差。
3. 满足“前端独立计算、不依赖后端逐次校验”的目标，同时保留抽检能力。
