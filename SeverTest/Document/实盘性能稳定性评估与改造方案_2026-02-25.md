# 实盘性能稳定性评估与改造方案（2026-02-25）

## 1. 背景与目标

本文档用于回答以下核心问题：

1. 当前实盘性能瓶颈在哪里，是否已经可以清晰定位。  
2. 当前架构是否可支撑 C 端量化产品的规模化增长。  
3. 当前分布式与多线程能力是否足够，下一步如何继续优化。  
d4. 如何基于数据做稳定性改造与验收。  

结论先行：

1. 当前主瓶颈仍是 `strategy.run.execute`。  
2. 实盘已支持分布式部署（核心节点 + 实盘节点）。  
3. 实盘已使用多线程多核心能力，但仍属于“部分并行”，还需要继续增强。  
4. 多线程多核心优化和分片调度优化都可行，且应并行推进。  

---

## 2. 当前架构与能力（最新）

### 2.1 服务器角色

当前角色已支持：

1. `core`：核心服务器（对外 HTTP/WS，含回测/实盘分发网关）。  
2. `backtest-worker`：回测算力服务器。  
3. `full`：全功能服务器。  
4. `live-trading-worker`：实盘服务器（连接核心节点，专注实盘链路）。  

### 2.2 实盘分布式链路

当前实盘分布式链路已接通：

1. 核心节点开放 `ws/live-worker` 接入。  
2. 实盘节点通过 `LiveTradingWorkerClientHostedService` 主动连接核心节点。  
3. 核心节点通过 `LiveTradingWorkerDispatchService` 下发 `upsert/remove`。  
4. 实盘节点执行策略增删并回传 `ack`。  
5. `StrategyOwnership` 租约机制继续保证单策略同一时刻仅一个节点执行。  

### 2.3 多线程多核心能力现状

当前是“外层保守 + 内层并行”模型：

1. `StrategyRuntimeHostedService` 外层策略消费固定 `1 worker`。  
2. `RealTimeStrategyEngine` 在匹配策略数 `>=8` 时启用 `Parallel.ForEach`。  
3. 引擎并行度默认取 `Environment.ProcessorCount`。  
4. `IndicatorEngine` 采用多 worker（默认约 `CPU/2`）。  

结论：当前并非单线程，但外层并发仍偏保守，热点叠加时仍会出现容量压力。

## 4. 问题定位与约束

### 4.1 已定位问题

1. 主瓶颈是 `strategy.run.execute`。  
2. `market.dispatch` 与 `action.enqueue` 不是主瓶颈。  
3. `trade.consume` 有秒级长尾，属于稳定性补强项。  

### 4.2 当前容量约束

1. 热点任务在同一分钟集中到达。  
2. 外层 `1 worker` 使热点叠加时总耗时累积明显。  
3. 分布式链路已具备，但调度仍有优化空间。  

风险判定（单节点/单热点分片均适用）：

`同一分钟内该执行单元待处理任务总耗时 > 60000ms`

## 6. 分阶段改造方案（最新）

### 6.1 P0：观测与告警固化（低风险，立即执行）

1. 固化每分钟任务总耗时（按交易所/币种/周期分组）。  
2. 固化热点键 TopN（按 `strategy.run` 总耗时）。  
3. 固化子阶段占比（lookup/indicator/execute）。  
4. 固化实例负载分布（按 `actor_instance`）。  
5. 预警线：单分钟总耗时 `>40s` 预警，`>55s` 告警。  

### 6.2 P1：核心执行优化（中风险，高收益）

1. 优先优化 `strategy.run.execute`：  
减少重复条件计算、减少临时对象分配、复用中间结果。  
2. 保持语义不变前提下做条件执行裁剪与批量化。  

### 6.3 P1.5：多线程多核心优化（重点）

1. 外层并发由固定 `1 worker` 升级为可配置并发。  
2. 并行阈值与最大并行度配置化（按核数与场景调优）。  
3. 引入热点键分片执行（同键串行、异键并行）。  
4. 增加并发保护（线程池监控、队列背压、超时降级）。  
5. 形成压测推荐矩阵（2核/4核/8核/16核）。  

### 6.4 P2：调度与分片升级（结构性优化）

1. 从“广播 + 租约竞争”演进到“定向分片分发”。  
2. 按 `exchange/symbol/timeframe` 或一致性哈希做策略归属。  
3. 在保证同键时序一致前提下提升多实例负载均衡。  

### 6.5 P3：交易消费长尾治理（稳定性补强）

1. 建立 `trade.consume` 长尾专项统计。  
2. 对请求、重试、超时做分层策略。  
3. 将慢单占比纳入稳定性 KPI。

## 8. 验收标准（必须量化）

1. 热点键（200策略）`strategy.run`：`P95 < 15s`。  
2. 单分钟总执行耗时：`P95 < 30s`，`P99 < 45s`。  
3. 跨分钟堆积率：`< 0.1%`。  
4. `trade.consume`：`P95 < 800ms`，`P99 < 2s`。  
5. 定位时效：`trace_id` 在 1 分钟内定位到阶段与实例。  
6. 分布式有效性：有多节点部署时，`actor_instance` 有效实例数 `>=2`。  
7. 并发有效性：CPU 利用率提升且无明显线程池饥饿/队列爆满。

```sql
       method, flow, duration_ms, error_message, metrics_json
FROM strategy_task_trace_log
WHERE trace_id = ?

```

```sql
-- 3) 最近15分钟按阶段统计吞吐与耗时
SELECT event_stage,
       COUNT(*) AS total,
       ROUND(COUNT(*)/900, 2) AS eps_15m,
       ROUND(AVG(COALESCE(duration_ms,0)), 2) AS avg_ms,
       MAX(COALESCE(duration_ms,0)) AS max_ms
FROM strategy_task_trace_log
WHERE created_at >= UTC_TIMESTAMP() - INTERVAL 15 MINUTE
GROUP BY event_stage
ORDER BY total DESC;
```

```sql
-- 4) 最近15分钟实例负载分布
SELECT actor_instance,
       event_stage,
       COUNT(*) AS total,
       ROUND(AVG(COALESCE(duration_ms,0)), 2) AS avg_ms,
       MAX(COALESCE(duration_ms,0)) AS max_ms
FROM strategy_task_trace_log
WHERE created_at >= UTC_TIMESTAMP() - INTERVAL 15 MINUTE
GROUP BY actor_instance, event_stage
ORDER BY actor_instance, total DESC;
```

```sql
-- 5) 最近15分钟 strategy.run P95/P99（MySQL 8）
WITH base AS (
    SELECT duration_ms
    FROM strategy_task_trace_log
    WHERE created_at >= UTC_TIMESTAMP() - INTERVAL 15 MINUTE
      AND event_stage = 'strategy.run'
      AND duration_ms IS NOT NULL
),
ordered AS (
    SELECT duration_ms,
           ROW_NUMBER() OVER (ORDER BY duration_ms) AS rn,
           COUNT(*) OVER () AS cnd
    FROM base
)
SELECT
    MIN(CASE WHEN rn >= CEIL(cnt * 0.95) THEN duration_ms END) AS p95_ms,
    MIN(CASE WHEN rn >= CEIL(cnt * 0.99) THEN duration_ms END) AS p99_ms,
    MAX(duration_ms) AS max_ms
FROM ordered;
```

```sql
-- 6) 热点键 strategy.run 子阶段
SELECT COUNT(*) AS cnt,
       ROUND(AVG(duration_ms), 2) AS avg_total_ms,
       ROUND(AVG(CAST(JSON_UNQUOTE(JSON_EXTRACT(metrics_json, '$.stage.lookupMs')) AS UNSIGNED)), 2) AS avg_lookup_ms,
       ROUND(AVG(CAST(JSON_UNQUOTE(JSON_EXTRACT(metrics_json, '$.stage.indicatorMs')) AS UNSIGNED)), 2) AS avg_indicator_ms,
       ROUND(AVG(CAST(JSON_UNQUOTE(JSON_EXTRACT(metrics_json, '$.stage.executeMs')) AS UNSIGNED)), 2) AS avg_execute_ms
FROM strategy_task_trace_log
WHERE created_at >= UTC_TIMESTAMP() - INTERVAL 15 MINUTE
  AND event_stage = 'strategy.run'
  AND exchange = 'binance' AND symbol = 'BTC/USDT' AND timeframe = '1m';
```
