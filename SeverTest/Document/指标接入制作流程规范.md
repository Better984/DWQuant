# 指标接入制作流程规范（简版）

## 1. 目的与适用范围
- 目的：统一指标接入方式，保证后续新增指标可快速落地、稳定运行、便于维护。
- 适用：贪婪恐慌、现货 ETF 流入流出、资金费率、持仓量等所有外部指标接入。

## 2. 总体原则
- 单指标单流程：每个指标按同一流程独立接入，不和其他指标混写业务逻辑。
- 分层清晰：Controller 只做参数与编排；业务在 Application；数据访问在 Infrastructure。
- 先文档后代码：先补协议与字段定义，再实现代码。
- 开发与生产隔离：开发期可用聚合商源，正式上线必须切换官方数据源。

## 3. 标准制作流程
1. 需求定义  
   - 明确指标名称、展示位置、更新频率、时间粒度、是否需要历史数据。
2. 协议设计  
   - 新增或复用协议 type，定义请求参数与响应字段（单位、精度、时间戳统一为毫秒）。
   - 同步定义 `config_json.fields[]` 字段目录（path/dataType/unit/conditionSupported/description）。
3. 数据模型与存储  
   - 定义统一模型（当前值 + 历史点位）。
   - 建表命名规范：`coinglass_{指标代码}_history`。
   - 示例：`coinglass.fear_greed` -> `coinglass_fear_greed_history`。
   - 必备字段：`id`、`coinglass_code`、`value`、`ts`、`source`、`created_at`。
4. 数据拉取与缓存  
   - 后台定时拉取（HostedService），失败重试并记录日志。
   - 先写数据库，再更新内存/Redis 缓存。
   - 去重规则：同指标同时间戳只保留一条。
   - 若有策略触发诉求，需在 payload 增加 `signals.*` 派生字段，并保证 latest/history 口径一致。
5. 服务对外提供  
   - 提供 latest + history 查询接口。
   - 统一 JWT 校验与协议包结构（Protocol Envelope）。
6. 前端接入展示  
   - 通过 network client 拉取，不直连第三方。
   - 显示单位、更新时间、空态与错误态。
7. 验证与回归  
   - 核对数据准确性（与上游对比）。
   - 验证更新频率、缓存命中、分页与时间范围查询。
8. 上线切换  
   - 切换 `CoinGlass.SourceMode` 到官方。
   - 检查 API Key、限频、告警策略后再发布。

## 4. 命名与结构规范
- 指标代码：`snake_case`（示例：`fear_greed`、`spot_etf_netflow`）。
- 来源类型：在定义 `config_json.source.type` 标识（示例：`http_pull`、`custom_compute`、`hybrid`）。
- 字段路径：统一用点路径（示例：`value`、`signals.netInflow3dAllPositive`）。
- 后端文件建议：
  - `Modules/Indicators/Application/{指标}Service.cs`
  - `Modules/Indicators/Infrastructure/{指标}Repository.cs`
  - `Modules/Indicators/Application/{指标}RefreshHostedService.cs`
- 前端文件建议：
  - `Client/src/network/indicator/{指标}Client.ts`
  - `Client/src/components/indicator/{指标}Card.tsx`

## 5. 日志与错误规范
- 日志必须中文，包含关键上下文：`indicatorCode`、`reqId`、`uid`（如有）。
- 错误分级：
  - 可重试错误：网络超时、上游 5xx。
  - 不可重试错误：参数错误、鉴权失败、字段结构不匹配。

## 6. 最小验收清单（上线前）
- 协议文档已更新。
- `indicator.meta.list` 字段目录可正确返回（fields 非空、conditionSupported 标注正确）。
- 建表脚本与模块说明已更新。
- `dotnet build`、前端 `npm run build` 通过。
- 乱码扫描通过。
- 开发源与生产源切换策略已确认。
