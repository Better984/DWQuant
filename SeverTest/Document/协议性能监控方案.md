# 协议性能监控方案

## 目标
- 记录每条协议交互的前端总耗时与后端处理耗时。
- 支持区分 HTTP 与 WebSocket 两类协议。
- 支持汇总统计、慢协议识别与异常分析，便于定位需要优化的协议。
- 支持通过 `appsettings.json -> Monitoring.EnableProtocolPerformanceStorage` 控制是否启用 MySQL 入库，默认关闭。

## 数据链路
1. 前端发起 HTTP / WebSocket 请求时生成或复用 `reqId`。
2. 后端在统一协议入口记录服务端开始时间，响应完成后写入服务端耗时。
3. 前端在收到响应后计算总往返耗时，并批量上报到后端。
4. 后端按 `reqId + transport` 将前后端数据合并到同一条记录。
5. 统计接口按协议类型聚合，输出平均耗时、最大耗时、慢请求占比与错误率。
6. 分析接口基于固定阈值和相对偏离规则标记异常协议。

> 当 `Monitoring.EnableProtocolPerformanceStorage=false` 时：
> - 不采集新的 HTTP / WebSocket 协议耗时；
> - 不启动后台写入任务；
> - 上报接口返回已关闭状态；
> - 汇总/分析接口返回空结果，避免误建表或继续写库。

## 采集口径
### HTTP
- 前端总耗时：浏览器发起请求到收到响应的总时间。
- 后端处理耗时：请求进入 ASP.NET Core 管道到响应完成写回的时间。

### WebSocket
- 前端总耗时：客户端发送请求消息到收到同 `reqId` 应答的总时间。
- 后端处理耗时：服务端完成消息解析后到应答发送完成的时间。
- `ping/pong` 心跳消息不纳入统计，避免污染业务分析结果。

## 存储表
- 表名：`monitor_protocol_performance`
- 唯一键：`req_id + transport`
- 关键字段：
  - `protocol_type`
  - `request_path`
  - `user_id`
  - `server_elapsed_ms`
  - `client_elapsed_ms`
  - `client_network_overhead_ms`
  - `protocol_code`
  - `http_status`
  - `is_success`
  - `is_timeout`

## 异常识别规则
- 固定阈值：
  - 服务端耗时 `>= 1000ms` 视为慢请求。
  - 前端总耗时 `>= 1500ms` 视为慢请求。
- 相对偏离：
  - 协议样本数 `>= 5` 时，若平均服务端耗时超过全局平均值 2 倍，则标记为相对异常。
  - 协议样本数 `>= 5` 时，若慢请求占比 `>= 20%`，则标记为高风险协议。
- 优先级判断：
  - 若 `server_elapsed_ms` 高，优先排查后端查询、聚合、锁竞争与序列化。
  - 若 `client_elapsed_ms` 高但 `server_elapsed_ms` 低，优先排查网络链路、浏览器主线程阻塞与大包传输。

## 输出接口
- `monitoring.protocol.performance.report`：前端批量上报耗时。
- `monitoring.protocol.performance.summary`：协议耗时汇总。
- `monitoring.protocol.performance.analyze`：异常协议分析结果。

## 落地原则
- 统一在协议入口层接入，避免每个业务接口重复埋点。
- 入库采用异步写入，降低业务请求额外开销。
- 日志、注释与返回字段统一使用中文说明，便于排查。
- 生产环境默认关闭协议性能入库，需要排查时再按配置显式开启。
