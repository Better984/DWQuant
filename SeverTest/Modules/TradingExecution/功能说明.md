# TradingExecution 模块功能说明

## 功能概述
- 策略动作消费与交易执行。
- 封装交易所下单能力（Ccxt）。
- 支持 `testing` 模式本地模拟执行：动作照常入队与消费，但不调用交易所 API，直接按行情价写入开/平仓记录。
- 开/平仓写库增加重试；当“已开仓下单成功但本地开仓写库失败”时，自动触发一次补偿平仓，降低交易所与本地状态分裂风险。
- 平仓写库失败时不再直接移除风控索引，只有确认仓位已写为 `Closed` 才执行后续内存状态清理。
- 开仓执行前改为“最大持仓数量”校验：信号照常触发，真正执行下单前按 `MaxPositionQty` 校验 `当前同向持仓 + 本次开仓量`，超限则阻断不下单。
- 平仓动作改为循环处理，若存在历史遗留同方向多仓，会连续平到无可平仓位为止（中途失败即中断并记录）。
- 新增订单主表与事件流水：每个策略动作都会先落 `trade_order_request/trade_order_event`，保留目标ID、归一化数量、最新状态、交易所订单号与恢复关联任务。
- 新增平台目标风控链：在下单前统一执行“基础参数校验 -> 市价数量归一化 -> 最大持仓限制”，只保留市价单，不引入限价/TWAP/VWAP。
- 恢复任务新增 `order_request_id` 关联，可在恢复成功/失败后同步回写订单主表状态（`recovery_pending/recovered/failed`）。
- 新增交易恢复队列：当“开仓补偿失败”或“平仓订单已提交但本地写库失败”时，任务会持久化到 `trade_recovery_task`，后台轮询重试；重试耗尽后发送 `Risk/Critical` 通知，形成可人工介入的闭环。
- 恢复任务支持“重启恢复”：服务启动时会回收遗留的 `processing` 任务并重新入队，避免进程异常退出造成恢复任务丢失。
- 消费前新增策略运行态二次校验：策略已卸载/已封禁时，自动丢弃残留动作，避免删除后队列尾包继续下单。
- 依赖注入生命周期规范化：`TradeRecoveryTaskRepository` 调整为 `Scoped`，`TradeActionConsumer` 通过 `IServiceScopeFactory` 按次创建作用域获取仓储，避免单例后台服务直接持有 `DatabaseService`（Scoped）。
- 开仓尝试记录：每次开仓成功/失败/上限阻断均写入 `order_open_attempt` 表，记录信号时间、信号价、上限值与阻断时持仓；`GetConsecutiveFailuresAsync(usId)` 仅统计真实下单失败，不把“上限阻断”计入连续失败。
- 连续失败自动暂停：连续开仓失败 3 次时，自动将策略状态设为 `paused_open_fail`，写入 `user_strategy` 与 `strategy_system_log`，并从运行时卸载；前端可明确展示「连续开仓失败已暂停」，用户可手动恢复为运行。

## 主要入口
- 消费与执行：`Application/TradeActionConsumer.cs`、`Infrastructure/CcxtOrderExecutor.cs`
- 目标风控：`Application/StrategyTargetRiskService.cs`
- 恢复入队：`Application/TradeRecoveryEnqueueService.cs`（供风控平仓、手动平仓等链路复用）
- 交易所适配：`Infrastructure/CcxtExchangeFactory.cs`、`Application/CcxtStrategyActionExecutor.cs`
- 订单流水：`Infrastructure/OrderLifecycleRepository.cs`

## 配置选项
- `TradingOptions`：交易沙盒模式等（`Trading:EnableSandboxMode` 控制是否启用交易所测试环境）。

## 联动关系
- `ExchangeApiKeys`：获取交易所凭证。
- `StrategyEngine`：接收策略动作与执行请求。
- `Positions`：更新仓位与平仓逻辑协同。
- `MarketStreaming`：使用行情/价格数据。
- `Notifications`：执行结果与异常通知。

## 变更记录
- 2026-03-06：新增 `trade_order_request/trade_order_event` 订单主表与事件流水，实盘动作改为“目标落库 -> 平台风控链 -> 市价单提交 -> 恢复关联”闭环；恢复任务新增 `order_request_id` 反向关联订单主表。
- 2026-02-24：开仓链路改为执行阶段按 `MaxPositionQty` 校验并阻断；新增上限阻断落库明细（信号时间/价格/持仓上下文），避免“命中信号但未开仓”不可追溯。
- 2026-02-23：连续开仓失败 3 次自动暂停策略（`paused_open_fail`），写入 `strategy_system_log`，前端可明确展示。
- 2026-02-23：新增 `order_open_attempt` 表与 `OrderOpenAttemptRepository`，开仓成功/失败均落库，支持连续失败次数查询。
- 2026-02-23：新增 `TradeRecoveryEnqueueService`，风控平仓、一键平仓、手动单仓平仓在「交易所成功但本地写库失败」时统一入队恢复任务，补齐状态分裂风险。
- 2026-02-23：新增交易执行写库重试与开仓失败补偿平仓逻辑，修复下单成功后本地状态未落库的核心分裂风险。
- 2026-02-23：修复重复开仓/平仓不对称，恢复防重复开仓并支持连续平仓处理同方向遗留多仓。
- 2026-02-23：新增交易恢复队列与失败告警闭环，覆盖“平仓写库失败”和“开仓补偿失败”两类高风险分裂场景。
- 2026-02-23：消费链路新增策略运行态校验，删除/卸载后的残留动作将自动丢弃。
- 2026-02-23：交易恢复队列改为数据库持久化（`trade_recovery_task`），支持重启后继续恢复执行。
- 2026-02-23：修复 `TradeRecoveryTaskRepository` 生命周期冲突（Scoped 仓储 + `IServiceScopeFactory` 作用域解析），解决启动期 `Cannot consume scoped service` 异常。
